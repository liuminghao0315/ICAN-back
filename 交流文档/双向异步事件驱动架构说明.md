# 双向异步事件驱动架构说明

## 架构概述

本系统实现了**双向异步事件驱动架构**，通过RabbitMQ实现Java和Python之间的解耦通信，使用Redis进行无状态的状态管理。

## 详细工作流程

### 阶段1：任务分发（Java → RabbitMQ → Python）

1. **用户触发分析**：前端调用Java API创建分析任务
2. **Java Producer**：
   - 标记任务为`PROCESSING`
   - 构造任务消息（包含`taskId`, `videoId`, `videoUrl`, `videoTitle`, `videoDuration`, `fileSize`）
   - 发送到RabbitMQ任务分发队列（`algorithm.task.queue`）
3. **Python Consumer**：
   - 监听任务分发队列
   - 收到任务后开始处理

### 阶段2：媒体分割与并发处理（Python内部）

1. **媒体分割**：
   - 模拟将视频文件分割为视觉流和音频流
   - 耗时：1-3秒（随机）

2. **并发启动两个线程**：
   - **Thread A（视觉流处理）**：
     - 分析视频画面内容（场景识别、人脸检测、质量评估等）
     - 耗时：3-8秒（随机）
     - 完成后发送`Video`类型消息到结果回调队列
   
   - **Thread B（音频流处理）**：
     - **Step 1：音频特征提取**
       - 提取声学特征（音质、语音比例、音乐比例、噪声水平等）
       - 耗时：2-5秒（随机）
       - 完成后发送`Audio`类型消息到结果回调队列
     - **Step 2：ASR转文本**
       - 将音频转换为文本
       - 分析文本内容（关键词、主题分类、情感分析等）
       - 耗时：2-6秒（随机）
       - 完成后发送`Text`类型消息到结果回调队列

3. **消息发送**：
   - Python异步发送3条独立消息（顺序不固定）
   - 消息格式：
     ```json
     {
       "taskId": "任务ID",
       "moduleType": "video|audio|text",
       "resultData": { /* 模块分析结果 */ }
     }
     ```

### 阶段3：状态聚合与进度计算（Java + Redis）

1. **Java Consumer接收消息**：
   - 监听结果回调队列（`algorithm.result.queue`）
   - 收到消息后解析`taskId`、`moduleType`、`resultData`

2. **Redis状态管理**：
   - **存储模块结果**：`analysis:results:{taskId}:{moduleType}` → `resultData`
   - **跟踪完成模块**：`analysis:modules:{taskId}` → Set（添加`moduleType`）
   - **查询完成数量**：`SCARD analysis:modules:{taskId}`

3. **动态进度计算**：
   - 1个模块完成 → 25%
   - 2个模块完成 → 50%
   - 3个模块完成 → 75%
   - 通过WebSocket推送进度到前端

### 阶段4：整合分析（Java内部）

1. **触发条件**：当Redis Set size达到3时（即video、audio、text都完成）

2. **整合分析执行**：
   - 从Redis获取所有模块结果
   - 生成受众年龄分布（`18-24`, `25-34`, `35-44`, `45+`）
   - 生成预测兴趣标签
   - 生成预测播放量
   - 生成预测互动率

3. **完成处理**：
   - 更新进度为100%
   - 保存完整分析结果到数据库
   - 标记任务为`COMPLETED`
   - 推送100%进度和整合结果到前端
   - **清理Redis数据**：删除所有相关Key

## 关键技术点

### 1. 线程安全
- **Redis Set操作**：`SADD`是原子操作，保证并发安全
- **进度计算**：基于Redis Set的`SCARD`操作，保证准确性

### 2. 无状态设计
- Java端不维护内存状态，所有状态存储在Redis中
- 支持多实例部署，实现水平扩展

### 3. 异步处理
- Python端使用多线程实现并发处理
- Java端通过RabbitMQ异步接收结果
- 前端通过WebSocket实时接收进度更新

### 4. 错误处理
- RabbitMQ消息手动确认机制
- 消息处理失败时重新入队
- Python端异常时发送错误消息


```json

  "taskId": "任务ID",
  "moduleType": "video|audio|text",

  }
```

```json
{
  "type": "task_progress",
  "message": "任务进度更新",
  "timestamp": 1704182400000,
  "data": {
    "taskId": "任务ID",

    "status": "PROCESSING|COMPLETED",

    "message": "视频分析完成",
    "moduleType": "video|audio|text|integration",
    "resultData": { /* 模块结果或整合结果 */ }
  }

```

## Redis数据结构

### Set: `analysis:modules:{taskId}`
存储已完成的模块类型
```
SADD analysis:modules:task123 "video"
SADD analysis:modules:task123 "audio"
SADD analysis:modules:task123 "text"
SCARD analysis:modules:task123  # 返回3
```

### String: `analysis:results:{taskId}:{moduleType}`
存储模块结果数据（JSON格式）
```
SET analysis:results:task123:video "{...}"
SET analysis:results:task123:audio "{...}"
SET analysis:results:task123:text "{...}"
```

### 过期时间
所有Key设置24小时过期时间，防止数据堆积

## 进度计算逻辑

| 已完成模块数 | Redis Set Size | 计算进度 | 说明 |
|------------|---------------|---------|------|
| 1 | 1 | 25% | 第一个模块完成（可能是video/audio/text中的任意一个） |
| 2 | 2 | 50% | 第二个模块完成 |
| 3 | 3 | 75% | 第三个模块完成 |
| 3 + 整合完成 | - | 100% | Java执行整合分析后 |

## 文件清单

### Python端
- `backend/algorithm_simulator.py` - 算法模拟脚本
  - `simulate_media_splitting()` - 模拟媒体分割
  - `process_visual_stream()` - 视觉流处理线程
  - `process_audio_stream()` - 音频流处理线程（顺序执行两个步骤）
  - `send_result_message()` - 发送结果消息

### Java端
- `backend/src/main/java/com/ican/project/service/impl/MockAlgorithmServiceImpl.java` - 任务生产者
  - `processTask()` - 发送任务到RabbitMQ
  
- `backend/src/main/java/com/ican/project/rabbitmq/AlgorithmResultListener.java` - 结果消费者
  - `handleAlgorithmResult()` - 监听结果队列，管理Redis状态
  - `performIntegrationAnalysis()` - 执行整合分析
  - `saveAnalysisResult()` - 保存分析结果

- `backend/src/main/java/com/ican/project/config/RabbitMQConfig.java` - RabbitMQ配置

## 测试验证

### 测试场景
1. **并发执行验证**：观察Python日志，确认两个线程在不同时间点完成
2. **消息顺序验证**：多次测试，确认3条消息到达Java的顺序不固定
3. **进度计算验证**：观察前端进度条，确认按25%→50%→75%→100%更新
4. **Redis状态验证**：使用Redis客户端查看Key，确认状态正确存储和清理
5. **整合分析验证**：确认当3个模块都完成时，自动触发整合分析

### 预期行为
- Python两个线程同时启动，随机延时完成
- Java收到消息的顺序不固定（取决于哪个模块先完成）
- 进度根据已完成的模块数量动态计算
- 当3个模块都完成时，自动触发整合分析
- 完成后Redis Key被清理

## 优势

1. **解耦**：Java和Python通过消息队列解耦，互不影响
2. **可扩展**：支持多实例部署，水平扩展
3. **容错**：消息持久化，支持重试机制
4. **实时性**：通过WebSocket实时推送进度
5. **无状态**：使用Redis管理状态，支持分布式部署
