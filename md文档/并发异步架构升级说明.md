# 并发异步架构升级说明

## 架构变更概述

将视频分析流程从**顺序执行**升级为**并发异步架构**，使用Redis进行状态管理，实现接近生产环境的架构。

## 核心变更点

### 1. Python端：并发执行
- **变更前**：按顺序执行音频→视频→文本分析
- **变更后**：使用多线程同时启动三个分析任务，随机延时完成

### 2. Java端：Redis状态管理
- **变更前**：使用内存Map存储模块结果
- **变更后**：使用Redis Set记录已完成模块，动态计算进度

### 3. 进度计算：动态化
- **变更前**：Python指定固定进度值
- **变更后**：Java根据Redis中已完成的模块数量动态计算

## 新的数据流

```
用户创建任务
    ↓
Java 发送任务到 RabbitMQ
    ↓
Python 获取任务
    ↓
┌─────────────────────────────────────┐
│ Python 并发启动三个线程              │
│ Thread 1: 音频分析 (随机2-8秒)      │
│ Thread 2: 视频分析 (随机2-8秒)      │
│ Thread 3: 文本分析 (随机2-8秒)      │
│ → 三个模块在不同时间点完成           │
│ → 随机顺序发送到 RabbitMQ            │
└─────────────────────────────────────┘
    ↓
Java 接收消息（顺序不固定）
    ↓
┌─────────────────────────────────────┐
│ Java Redis 状态管理                  │
│ 1. 存储模块结果到 Redis              │
│ 2. 将模块类型添加到 Redis Set        │
│ 3. 查询已完成模块数量                │
│    - 1个模块 → 25%                   │
│    - 2个模块 → 50%                   │
│    - 3个模块 → 75%                   │
│ 4. 推送进度到 WebSocket              │
└─────────────────────────────────────┘
    ↓
检测到3个模块都完成
    ↓
┌─────────────────────────────────────┐
│ Java 触发整合分析（第4步）            │
│ 1. 从Redis获取所有模块结果            │
│ 2. 生成受众年龄分布                   │
│ 3. 推送100%进度                      │
│ 4. 清理Redis Key                     │
└─────────────────────────────────────┘
    ↓
Vue3 前端展示
```

## 代码修改清单

### Python脚本 (`algorithm_simulator.py`)

**主要变更：**
1. 导入 `threading` 模块
2. 修改 `send_progress_message`：移除 `progress` 参数
3. 新增 `process_module` 方法：在独立线程中执行单个模块分析
4. 修改 `process_task` 方法：使用多线程并发启动三个分析任务

**关键代码：**
```python
# 创建三个线程，并发执行
audio_thread = threading.Thread(
    target=self.process_module,
    args=(task_id, task_message, "audio", self.simulate_audio_analysis, "音频")
)
video_thread = threading.Thread(...)
text_thread = threading.Thread(...)

# 同时启动所有线程
for thread in threads:
    thread.start()
```

### Java监听器 (`AlgorithmResultListener.java`)

**主要变更：**
1. 注入 `RedisTemplate`
2. 移除内存 `ConcurrentHashMap`，改用Redis
3. 使用Redis Set记录已完成的模块
4. 动态计算进度（根据Redis中模块数量）
5. 从Redis获取所有模块结果进行整合分析
6. 完成后清理Redis Key

**Redis Key设计：**
- `analysis:modules:{taskId}` - Set类型，存储已完成的模块类型
- `analysis:results:{taskId}:{moduleType}` - 存储模块结果数据

**关键逻辑：**
```java
// 记录模块完成
redisTemplate.opsForSet().add(modulesKey, moduleType);

// 查询已完成数量
Long completedCount = redisTemplate.opsForSet().size(modulesKey);

// 动态计算进度
int progress = (int) (completedCount * 25); // 1个=25%, 2个=50%, 3个=75%

// 检测到3个模块都完成
if (completedCount == 3) {
    // 从Redis获取所有结果
    // 执行整合分析
    // 清理Redis Key
}
```

## 消息格式变更

### Python发送的消息（新格式）
```json
{
  "taskId": "任务ID",
  "moduleType": "audio|video|text",
  "resultData": {
    // 模块分析结果
  }
}
```
**注意**：不再包含 `progress` 字段

### Java推送的进度消息
```json
{
  "type": "task_progress",
  "message": "任务进度更新",
  "timestamp": 1704182400000,
  "data": {
    "taskId": "任务ID",
    "videoId": "视频ID",
    "status": "PROCESSING",
    "progress": 25|50|75|100,  // 由Java动态计算
    "message": "音频分析完成",
    "moduleType": "audio|video|text|integration",
    "resultData": { ... }
  }
}
```

## 进度计算逻辑

| 已完成模块数 | 计算进度 | 说明 |
|------------|---------|------|
| 1 | 25% | 第一个模块完成 |
| 2 | 50% | 第二个模块完成 |
| 3 | 75% | 第三个模块完成 |
| 3 + 整合完成 | 100% | Java执行整合分析后 |

## Redis数据结构

### Set: `analysis:modules:{taskId}`
存储已完成的模块类型，例如：
```
SADD analysis:modules:task123 "audio"
SADD analysis:modules:task123 "video"
SADD analysis:modules:task123 "text"
SCARD analysis:modules:task123  # 返回3
```

### String: `analysis:results:{taskId}:{moduleType}`
存储模块结果数据（JSON格式）

## 并发安全性

1. **Redis操作**：使用Redis Set的原子操作，确保并发安全
2. **进度计算**：基于Redis Set的size操作，保证准确性
3. **整合分析触发**：使用 `completedCount == 3` 判断，确保只触发一次

## 测试验证

### 测试场景
1. **并发执行验证**：观察Python日志，确认三个模块在不同时间点完成
2. **随机顺序验证**：多次测试，确认模块完成顺序不固定
3. **进度计算验证**：观察前端进度条，确认按25%→50%→75%→100%更新
4. **Redis状态验证**：使用Redis客户端查看Key，确认状态正确存储和清理

### 预期行为
- Python三个线程同时启动，随机延时完成
- Java收到消息的顺序不固定（取决于哪个模块先完成）
- 进度根据已完成的模块数量动态计算
- 当3个模块都完成时，自动触发整合分析
- 完成后Redis Key被清理

## 注意事项

1. **Redis连接**：确保Redis服务正常运行
2. **Key过期**：设置了24小时过期时间，防止数据堆积
3. **错误处理**：如果某个模块失败，不会影响其他模块的执行
4. **幂等性**：Redis Set的add操作是幂等的，重复添加同一模块不会影响结果
